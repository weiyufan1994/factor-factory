# 私募级 Factor Factory 缺口路线图（2026-04-14）

## 目的

本文用于记录当前系统在以下两者之间的真实差距：
- **已经存在的、由 CPV 样例支撑的 Step1–Step5 最小可复现链路**；
- **真正达到私募级标准的因子工厂**，即具备泛化、可重复、可生产、可投资使用能力的系统。

本文作为仓库内保留的正式路径文档，供后续工程化与研究硬化使用。

## 当前真实状态

目前仓库可以诚实地主张：
- Step1–Step5 的仓库骨架已成立；
- CPV 样例已证明“从研报到因子闭环”的真实路径可以跑通；
- contracts / fixtures / sample runners / handoff objects / 双语文档已进入仓库。

但仓库还不能诚实地主张：
- 已达到私募级泛化因子生产；
- 已实现跨因子的稳定标准化；
- 已具备工业级回测与评估治理；
- 已达到完整可投资部署状态。

## 定义：什么叫“私募级因子工厂”

私募级因子工厂，不是“能跑一个漂亮样例”的系统。
它至少必须同时满足以下四个条件：

1. **可泛化（Generalizable）**
   - 不止 CPV，而是多个因子家族都能以稳定语义通过流水线。
2. **可重复（Repeatable）**
   - 在相同输入下，多次运行的边界稳定、来源清楚、输出可解释。
3. **可生产（Production-capable）**
   - 系统支持批量处理、任务编排、重试和失败治理，而不需要人工盯着单个样例 babysit。
4. **可投资（Investment-usable）**
   - 输出可以进入真实研究、评估、组合与风控流程，而不是停留在研究玩具阶段。

所以，私募级要求本质上是以下能力的联合体：
- 研究抽取能力；
- 数据/适配器工程能力；
- 回测/评估能力；
- 治理/审计能力。

## 当前已经具备的东西

当前仓库已经具备：
- Step1–Step5 的最小链路；
- CPV 样例闭环；
- handoff / contracts / docs / fixtures / runners；
- 整理过的仓库结构与命名原则；
- 中英文双语仓库文档。

这意味着：**工厂骨架已经有了，样板产线也已经有了。**
当前真正缺的是规模化、稳定化、标准化，以及投资级硬化。

## 12 项硬缺口

### 1. 多样例泛化矩阵
**当前状态**
- CPV 是最主要的证明点；
- 其他因子家族尚未形成系统性通过记录。

**私募级要求**
需要建立一个真实样例矩阵，例如覆盖：
- 高频价量类；
- 资金流类；
- 基本面类；
- 另类数据类；
- 事件驱动类。

**差距**
当前仓库证明的是“这条路能走”，还没有证明“这条路在一类又一类因子上都能走”。

### 2. Step 1 抽取稳定性与误差边界测量
**当前状态**
- Step 1 已能从研报中抽取 factor idea；
- 但 prompt / route / chief merge 的表现仍然对模型较敏感。

**私募级要求**
需要显式测量：
- 哪类研报容易错；
- 哪类公式常漏；
- 哪些逻辑是 native，哪些只是 inferred；
- 错误率 / 歧义率 / 纠正率是多少。

**差距**
现在的 Step 1 更像一个强力研究助手层，还不是一个边界清楚、受控的生产级抽取模块。

### 3. Step 2 spec 标准化
**当前状态**
- Step 2 可以把 alpha idea 变成 factor spec；
- 但跨步骤对象依赖与样例式修补仍然存在。

**私募级要求**
需要一个稳定的 `factor_spec_master` 标准，覆盖：
- 输入字段定义；
- 计算窗口；
- 归一化口径；
- 缺失值策略；
- 滞后策略；
- 调仓频率；
- 可交易性边界。

**差距**
现在 schema 已存在，但“跨因子稳定标准”尚未被证明。

### 4. Step 3 数据映射与实现标准件化
**当前状态**
- Step 3 已能为 CPV 找到数据源、生成 qlib-friendly adapter，并给出首版实现。

**私募级要求**
需要统一的 adapter 标准层，覆盖：
- 日频 / 分钟频 / 基本面 / 另类数据；
- 字段命名规范；
- instrument / datetime 语义统一；
- 数据源切换时，上层逻辑不需要重写。

**差距**
当前 Step 3 仍带有“围绕样例成功而优化”的痕迹，距离通用 adapter 平台还差一层工程化。

### 5. Step 4 回测引擎工业化
**当前状态**
- Step 4 已能跑轻量样例路径，并输出 partial / diagnostics。

**私募级要求**
需要稳定具备：
- 标准回测接口；
- 成本模型；
- 滑点模型；
- 调仓逻辑；
- 行业 / 市值 / 风格中性处理；
- long-short 与分层评估；
- OOS / walk-forward / rolling validation。

**差距**
当前系统已经“能跑回测”，但还不是“私募研究标准回测平台”。

### 6. Step 5 评估体系仍不够投资化
**当前状态**
- Step 5 已能完成 factor case 的收口与评估输出。

**私募级要求**
需要投资可用的评价标准，例如：
- IC / RankIC / ICIR；
- 分层收益；
- 多空收益；
- 换手 / 容量 / 冲击成本代理；
- 风格暴露；
- 行业集中度；
- 回撤与稳健性；
- 生命周期监控。

**差距**
现在的 Step 5 更像研究收口层，而不是投资委员会可直接使用的决策层。

### 7. 因子库与版本治理
**当前状态**
- 单个 factor case 的闭环已经存在；
- 但完整的因子资产管理层尚未建立。

**私募级要求**
需要：
- 因子注册表；
- 因子版本号；
- 变更历史；
- 因子谱系；
- freeze / deprecate 规则；
- 同类因子的去重与归并规则。

**差距**
当前状态更像“我们能做出一个因子”，还不是“我们能经营一座因子库”。

### 8. 质量控制与失败分流规则
**当前状态**
- truthful partial 纪律已经建立。

**私募级要求**
仍需要更强的 QC 规则，例如：
- 哪些环节允许自动降级；
- 哪些环节必须硬中止；
- 哪些输出必须人工复核；
- 哪些缺失字段使下游无效；
- 如何识别 hallucinated specs / fake logic / 不可交易实现。

**差距**
当前系统已经有诚实纪律，但还没有完全形成工厂级规则体系。

### 9. 批量调度与任务编排
**当前状态**
- 目前仍主要是单因子 / 单样例式使用。

**私募级要求**
需要：
- 多篇研报 intake 调度；
- 多个 alpha idea 并发进入 Step2/3；
- 重试；
- 队列与优先级；
- 失败告警；
- 汇总台账与运行记账。

**差距**
现在更像“人工导演一条产线”，还不是“车间自动化生产”。

### 10. 成本、时延与吞吐量管理
**当前状态**
- 样例能跑，但成本/时延剖面尚未正式化。

**私募级要求**
需要知道：
- 每篇研报抽取成本；
- 每个因子构建成本；
- 回测耗时；
- 哪一层最贵；
- 哪一层最容易失败；
- 如何做缓存与复用。

**差距**
当前重心仍是“先做出来”，还不是“可运营地长期跑”。

### 11. 人工覆核与投研工作流整合
**当前状态**
- chief merge 与 truthful partial 已体现治理意图。

**私募级要求**
仍需要明确的人类工作流支持：
- 复核节点；
- 批注与修订；
- spec override；
- pass / defer / reject 标记；
- 研究员与 PM 的交互闭环。

**差距**
当前系统能自闭一些回路，但还没真正嵌入真实投研组织。

### 12. 从研究结果到组合决策的最后一公里
**当前状态**
- 实务上的终点经常仍是 `factor_case_master`。

**私募级要求**
需要接上：
- 因子池管理；
- 因子组合与优化；
- 风险模型交互；
- 与当前持仓 / 主题 / 风格的关系；
- 分阶段部署；
- 上线前监控。

**差距**
当前终点是“研究完成”，不是“投资部署完成”。

## 这 12 项背后的 4 个大块

如果把问题压到最高层，仓库实际上仍缺四大能力块：

### A. 泛化能力
需要证明工厂不是 CPV-only。

### B. 工业能力
需要稳定、可批量、低故障地运行，而不是单次样例成功。

### C. 投资能力
需要产出能被研究委员会 / PM 工作流直接消费的结果。

### D. 治理能力
需要可审计、可维护、可强制执行、可版本化的输出，而不只是“看起来合理”的输出。

## 当前定位

当前最准确的定位是：

> **Factor Factory v0.1：样例闭环已成立，仓库骨架已搭好，CPV 已证明从研报到因子闭环的真实路径存在。**
> 但它还不是一个私募级、批量化、标准化、可投资部署的因子工厂。

## 下一阶段的优先建设顺序

如果资源有限，下一阶段最值得优先投入的四项是：

### Priority 1 — 多样例泛化矩阵
先证明工厂不是只会 CPV。

### Priority 2 — Step 3/4 标准化
统一 adapter 语义与回测接口。

### Priority 3 — Step 5 投资级评估
把研究闭环升级为投资可用评估闭环。

### Priority 4 — 因子库与版本治理
让工厂能积累可复用的因子资产，而不是每次都从头重建。

## 建议的分期路线

### P0 —— 从样例成功走向结构化可重复
- 增加 3–5 个非 CPV 因子样例；
- 强化 Step1 / Step2 的 schema 与歧义测量；
- 稳定 Step3 adapter 语义；
- 稳定 Step4 quick path 与 native path 边界；
- 统一 Step5 evaluation payload 字段。

### P1 —— 从可重复走向生产工作流
- 批量调度 / 重试 / 台账；
- 因子注册表与版本治理；
- QC 与失败分流规则；
- 成本 / 时延 / 限流画像；
- 人工复核节点。

### P2 —— 从生产工作流走向私募级工厂
- 面向组合的评估标准；
- 因子池接入；
- 风险模型与暴露控制；
- 分阶段部署路径；
- 生命周期监控与退役规则。

## 一句话总结

正确的下一目标，不是单纯“再多几个文件”或“再跑几个样例”，而是把 source code、fixtures、回测标准、评估标准、治理机制与面向组合的输出一起补强，直到这套系统在 CPV 之外也能作为可信的因子工厂存在。
